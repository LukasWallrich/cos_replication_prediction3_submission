Can a feature model predict health replication?

COS Prediction Challenge — Round 3 · investigation into the feature slot (m1)

Analysis run 2026-05-29 · reproduce with Rscript pipeline/19_health_transfer_model.R and the scratch_* diagnostics referenced below.

Bottom line

No tabular feature model reliably beats the base rate on challenge-round health — the kind of data Round 3 actually is. Across every design the challenge-health Brier sits at the base rate (~0.25); the best curated feature set only edges just below (0.244), while the LLM ensemble reaches 0.217. The one lever that helps is curating features for transferability (drop the sign-flipping r_harmonized, add design type); re-weighting training rows — up or down — does not. This is a structural ceiling, not a tuning problem.

  • The "feature model is fine on health" belief was an artifact of mixing two different populations: easy in-distribution FORRT health rows (small studies) with the challenge-round rows that actually resemble R3 (large studies).
  • A core feature of the deployed model — r_harmonized (effect magnitude) — flips sign between psychology and health, so the model applies it backwards on R3.
  • The feature slot's real value in best-of-3 is decorrelation, not accuracy. A health-aware / design-feature model is the most decorrelated from the LLM we found (R3 corr 0.34 vs the deployed 0.59) — that is what it's good for.

1 · Why the feature slot matters

Round 3 is scored on the best of three prediction columns, so the goal is three decorrelated bets. Two of the three current slots are LLM-derived. The feature slot is the only signal orthogonal to the LLM — it protects the floor if the LLM is systematically wrong on health. So we want it (a) as decorrelated from the LLM as possible and (b) as good as it can honestly be on health.

The hard constraint: R3 is 100% health, but the labelled training data (FORRT + R1 + R2) is ~1% health and mostly psychology. The feature model must therefore extrapolate into a regime it barely saw.

2 · Why the deployed model struggles on health

2.1  Random forest can't extrapolate

The deployed m1 is a fair-meta stack on the 32-feature enriched set, ~0.82 random forest. An RF prediction is the mean outcome of the training rows in the same leaves; past the largest split it ever learned, every point drops into the same terminal leaf and gets one flat value. Health n_o reaches ~55,000 vs mostly small-n psychology training, so RF flattens across the whole large-n health regime. Empirically its health discrimination is stuck at Spearman ≈ 0.25 regardless of how many features it is given. A linear/GAM term in log(n) can project a relationship into the unseen range — but only helps if the relationship is real and transferable (a GAM on the full enriched set extrapolated psychology-specific relationships and blew up to 0.308 on the health holdout).

2.2  It is not a "too many / too few features" problem

Of 437 evaluable features, 152 have |Spearman| > 0.15 with the health outcome — health signal is widespread. But of those 152, ~100 point the same way in psychology and health (transferable) and ~52 point the opposite way (anti-transfer). Those 52 poison any parametric learner trained on psychology and applied to health.

Sweeping the feature-set size confirms the lever is transferability, not count. On the leave-all-health-out holdout (n=102, base-rate Brier 0.2496):

2.3  The decisive split: FORRT-health vs challenge-health are different populations

The 102 labelled health rows are not interchangeable. 55 come from the FORRT replication database; 47 are challenge-round claims (R1/R2). R3 matches the challenge rows on every measurable axis:

The driver is sample size: FORRT-health is small lab-scale studies (median n=81); challenge-health and R3 are large epidemiological/survey studies (median n≈600–1,900). Training is dominated by small-n studies, so the model learns the small-n regime and extrapolates poorly into the large-n regime R3 occupies. Both populations have ~0.49 base rate — so this is not "FORRT is easier to call," it is a genuine sample-size covariate shift, and R3 is on the challenge side of it.

Two findings that change the picture

  • n_o transfers cleanly (+0.37 in both regimes) — sample size does not lose power at large n; it's the strongest stable predictor. What loses power at large n is the significance statistic (pval_z: +0.54 → +0.13), exactly as expected when "everything is significant."
  • r_harmonized anti-transfers: in small-n psychology a bigger effect → more replicable (+0.38, winner's-curse-free); in large-n health a bigger reported effect → less replicable (−0.35) — it's the surprising/inflated ones (note surprisingness is the single strongest health predictor at −0.38). The deployed model, trained mostly on psychology, applies this backwards on R3. evidence_strength_score (which multiplies in |r_harmonized|) goes dead for the same reason.

Design type is the most transferable health signal (observational vs experimental/intervention, surprisingness, sample adequacy) — all ~100% covered on R3, all same-sign in FORRT — and none of it is in the deployed enriched / stable_top sets.

4.3  Re-weighting the training rows — up or down — does not help

Given that the lever is feature choice, does re-weighting rows add anything? On the curated set, neither direction helps on challenge-health:

Re-weighting would help only if the off-distribution rows carried a different feature→outcome relationship that contaminates the fit — but that contamination lived in the anti-transfer features (chiefly r_harmonized), which the curated set already drops. Once every feature transfers, small-n and large-n rows estimate the same relationships, so downweighting the small-n rows just discards effective sample size (710 → 190) for nothing. Selecting transferable features and re-weighting rows attack the same covariate shift — and feature selection wins, because it removes the bias without costing sample size. We stopped here; pushing further on 47 rows would be overfitting the holdout.

5 · Methodology & design questions

5.1  Why the numbers are trustworthy (no circular selection)

The transferability screen needs the health outcomes, so naïvely "select on all 102 health rows then score on them" would be circular. Two leakage-free evaluations avoid this: (A) repeated grouped 5-fold CV with feature selection inside each fold (out-of-fold health rows never seen during selection or fitting); (B) leave-all-health-out, with selection on psychology only. R3 outcomes are never used (they don't exist). The screen uses only train outcomes (allowed), test feature distributions (allowed), and training-fold health sign-agreement.

5.2  Sample size, significance, and the "replication = significance in the same direction" rule

Because the outcome is the replication achieving significance in the same direction, a large original n is good signal — a precise, less winner's-cursed estimate that is more likely to be real and replicate (consistent with n_o's +0.37 in both regimes). What degrades at large n is the original study's significance statistic: when everything is significant, pval_z loses variance and discriminative power (+0.54 → +0.13).

5.3  The SE / precision conversion — worth it for linear models?

Yes, with a caveat. Two distinct things:

  • SE on its own ≈ 1/√n (for a correlation, on the Fisher-z scale literally 1/√(n−3)) → redundant with n. A standalone SE or CI-width feature adds little.
  • The precision-weighted effect = effect ÷ SE = the test statistic (z/t) = the effect × √n interaction. A linear model cannot form this from {effect, n} (it can only add them); a tree can build it by splitting. So handing a linear learner the precision conversion explicitly does add representational power. This is the case for keeping it.

Practically: we already include these conversions — pval_z, t_proxy_from_r_h, snr_proxy_h, evidence_strength_score — and they clearly earn their place in the psychology regime (FORRT-health: power_proxy +0.57, pval_z +0.54, evidence_strength +0.49). But on health they largely collapse: the z-statistic saturates at large n, and the r-derived versions inherit r_harmonized's sign-flip (evidence_strength → −0.05). Recommendation: keep a precision term for the linear learners (it's cheap and already present); for the health slot prefer the p-value-based one (pval_z, right sign) and drop the r-magnitude-based conversions, which can mislead. None of it breaks the base-rate ceiling, so it is a second-order tuning point.

5.4  Weighting the feature-slot ensemble — and an open design choice for the team

The feature slot (m1) is an equal-weight ensemble of RF / GAM / ElNet / XGB over the curated set, fit unweighted. Two different "weighting" questions came up; they have different answers, and the second is an open decision worth discussing.

(a) Sample-weighting (IPW) — tested, doesn't help once features transfer. The previous design up-weighted training rows resembling R3 (large-n, recent) via a 14-covariate inverse-propensity weight, applied to the parametric learners. Re-run on the curated set, the real IPW is neutral-to-harmful for every learner (ΔBrier vs unweighted, challenge-health): RF +0.003, GAM +0.007, ElNet +0.005, XGB +0.002; the blend goes 0.238 (unweighted) → 0.241 (parametric-IPW) → 0.242 (all-IPW). This reconciles with the earlier finding that IPW helped the parametric learners on the enriched set: IPW corrects covariate shift by re-focusing a mis-specified model on the target region, and enriched contains features whose relationship to the outcome shifts between the psychology-train region and the R3-like region. The curated set was chosen for transferability — its relationships are invariant across the distribution by construction — so there is no shift for IPW to correct, and it only spends effective sample size. IPW and feature-transferability are substitutes; once the features transfer, sample-weighting is not the lever.

(b) Learner-weighting — the open suggestion. The per-learner numbers expose something more useful. On challenge-round health the extrapolating learners clearly beat the tree learners:

learnerchallenge-health Brierextrapolates to large-n?
ElNet0.224yes — linear in log-n

This is the §2.1 extrapolation point showing up inside the ensemble. The tree learners (RF, XGB) can only average nearby training rows, so in R3's large-n regime they flatten and sit at the base rate; the linear/additive learners (ElNet, GAM) project the sample-size relationship into that regime and do better. The equal-weight blend is dragged up by the two trees. Tilting the blend toward ElNet/GAM (e.g. RF and XGB at half weight, or ElNet+GAM only) would likely move the feature slot from ~0.238 toward ~0.230 on challenge-health — and it is mechanistically motivated (the same extrapolation argument that runs through this whole analysis), not curve-fitting.

Open decision for the team — how to weight the feature-slot ensemble

  • Keep equal-weight (as deployed): most robust, no tuning — but knowingly includes two learners (RF, XGB) that can't handle R3's large-n regime.
  • Fixed tilt toward ElNet/GAM (e.g. RF/XGB at half weight): mechanistically justified by the extrapolation diagnosis; ~0.008 better on challenge-health in-sample.
  • ElNet + GAM only: the cleanest expression of "use only the learners that can extrapolate," but drops the trees' diversification and leans on two correlated linear models.

Caveat: n = 47, so the ~0.008 gap is within the noise band — treat this as a judgment call, and use a fixed rule rather than a learned weight to avoid overfitting the holdout. The best-of-3 E[min] barely moves either way (the raw LLM carries ~93% of the win-share regardless); this choice is about making the training-data slot as good as it honestly can be, not about the headline number. Sample-weighting (IPW) is settled — it doesn't help the curated set; learner-weighting is the live question.

6 · Conclusions

  • The feature model's ceiling on R3-like (large-n, challenge-round) health is the base rate — robust across every design tried. At large n, significance is uninformative and the replication signal lives in effect magnitude/precision, design, and bias — which tabular features capture weakly (and R3's effect sizes are 27% synthesised, i.e. noisy). The LLM wins because it integrates these holistically; a 47-row-validated tabular model cannot.
  • Feature transferability is the only lever that moved the number — the curated transferable set (drop r_harmonized, add design type) is the best feature model (0.244, just below the base rate). Re-weighting the training rows — upweighting health or downweighting off-distribution data — does not help: once the features transfer, all rows estimate the same relationships, so reweighting only trades away sample size.
  • The deployed model has a concrete defect on health: r_harmonized / evidence_strength_score anti-transfer, so it applies effect-size in the wrong direction on R3 — likely why m1 sits below the base rate on the n=44 health gate.
  • The feature slot's job is decorrelation, not accuracy. The health-aware / design-feature model is the most decorrelated from the LLM we found (R3 corr 0.34 vs deployed 0.59) — that is its real value in best-of-3.
  • Design type (and surprisingness, sample adequacy) is worth adding — it is the most transferable health signal and improves decorrelation — even though it does not lift the slot past the base rate.

7 · Recommendations

  • Feature slot: use the curated transferable set (n_o, surprisingness, sample adequacy, design type, n_f_tests) — best feature model and most decorrelated from the LLM (corr 0.34–0.51) — and at minimum drop or sign-correct r_harmonized / evidence_strength_score on the health slot. Don't expect it to beat the base rate on R3.
  • Gate & report on challenge-round health (R1/R2, n=47), not all 102 — update 18_health_gate.R, which currently flatters m1 via the FORRT rows.
  • Portfolio: the LLM is the only signal that beats the base rate on R3-like health, so it should carry the health bet; the feature slot is the orthogonal hedge. (This reinforces the separate recommendation that the dominated bridge m2 slot become a second LLM bet.)
  • If pursuing further feature gains: the ceiling says they won't come from new learners — only from new signal aimed at the large-n regime (richer LLM-extracted study descriptors), and even that is data-starved at n=47.

Caveats

  • n = 47 challenge-health rows. CIs are wide (~±0.03); differences between feature models are within noise. Treat them as clustered at the base rate.
  • Challenge-health is the best available R3 analog (matches on sample size, era, ES quality) but still imperfect — different rounds, journals, and sampling seed.
  • The automated screens are permissive at ~80 training-health rows, so they still admit some non-mechanism features. A hand-curated mechanism set performs within noise and is more defensible.
